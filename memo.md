# dotfiles分析

フレームワークとして確立できないか、ケース分けして考える。

* dotfilesの目的
    * 新しく買ったPCで初期セットアップ
    * OS再インストール後のセットアップ
    * SSH先で快適な環境を整える
    * 設定のバージョンアップを反映
    * tmuxでバージョンにより大きく設定ファイルの書き方変わるんだけどなんとかして欲しい
    * powerline入っている/入っていない/入れられないもよしなにして欲しい
    * 何度実行しても安全でいて欲しい
    * 時間はある意味いくらかかってもいいが、再実行性や元に戻すことが気軽にできないのは困る
    * linux/Mac/Windowsで設定をできる範囲で共有して欲しい
        * ある程度無理があるのは承知。つまり環境やバージョンで分岐せざるを得ない。それはしょうがないと受け入れよう
        * 逆にそれで分岐しなくていいような人はたぶんこのフレームワークいらない。会社でmacを使いsshでcentos6に入り自宅でubuntu最新版を使う僕みたいな人間のみが必要
    * [sshrc](http://qiita.com/ikuwow/items/ba4ca57fd67c06fd1b19)(ssh先に設定ファイルや設定の適用を行う仕組み)も見たが、わりと用途が限定されている気がする。割りきってどこでもdotfilesをセットアップしたほうがよいかな
* 現行必要だと思える機能は2つ
    * 一つはdotfilesの有名なqiita記事にかかれているようにビルドとデプロイを分けること。ビルドが毎回走る時間の無駄を省くため
        * 本当か？両者は本来同じことでは？時間的な問題は度外視するのが基本方針だからそれらは考えなくてよいのでは？
    * もう1点必要なのはその環境(OSだけでいいか？)とアプリケーションのバージョンのチェック機能
        * 入っていない機能を入れてもしょうがないし、バージョンが違うと設定ファイルのフォーマットが変わっており不正になる可能性がある
            * 特に新しい設定を古いアプリケーションへ適応しようとすると問題が発生しやすい
            * バージョンの認識機能と合わせる形で、入っていないアプリケーションの設定は入れないようにすれば良いと思う
    * すでにある設定ファイルはどうする？例えば.zshrcなどはデフォルトである程度作られる。一律削除は流石にまずいが、リネームしてバックアップするような形式にするか？
    * gitconfigのように会社とプライベートで分けたいようなファイルもある。そのへん勘案するとバージョンによる違いだけじゃなくて他の条件についても分岐できるべき？
    * 最悪元の状態に戻せない、という不安は結構つらい。適用に失敗したらその環境での作業がぶっ壊れるわけで、不安は大きい
        * しかし可逆性を保証することはかなり難しい。
        * セットに対象ファイル一覧リストを含めたらどうか？適用前に別の場所へ退避させる。これなら必ず最初から適用できるので適用時の状態が一貫しない問題は発生しない
            * しかし必然として毎回ビルドする手間は発生するが許容範囲か？
    * software linkで済ませるべきかも疑問
        * software linkを使う理由はリポジトリ側の更新がgit pullだけで伝えられるから。それもそれビルドとデプロイを同一視する場合は歪んでいない？
        * software linkが必ずしも使えないという事情もある。
            * 例えばsshのファイルは実ファイルであるひつようがある、など
            * かなり例外的ではある
            * そもそも意図していないはずだが、概ねlinkで正常に動いているのも事実

## 上記構造をまとめた事実

* そのアプリケーションについて関係のあるファイル・ディレクトリの一覧
* バージョン・環境ごとの識別方法
* バージョンごとのビルド・デプロイ方法(一旦その両者はわけない)
